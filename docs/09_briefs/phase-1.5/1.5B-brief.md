## 1. Goal

**Phase 1.5B lands user profiles, the invitation-based membership
lifecycle, per-org MFA enforcement, and login/logout auditing so
Phase 1.2 (agent integration) has a complete identity + permissions
surface to build against.**

What "done" means in one paragraph: every Supabase Auth user has an
auto-created `user_profiles` row with name, avatar, locale, timezone,
and `last_login_at`. Controllers can invite new users to an org via
`inviteUser` (which returns a crypto token for manual sharing — email
delivery is Phase 2). The invitee accepts via `acceptInvitation`
(bcrypt-compared token), which creates an active membership and
marks the invitation accepted. Memberships carry a `status` column
(`active`, `invited`, `suspended`, `removed`) and an `is_org_owner`
flag (partial unique — one per org). Suspended/removed users are
invisible to operational queries (RLS helpers, `buildServiceContext`,
`canUserPerformAction` all filter to `status = 'active'`). Orgs
with `mfa_required = true` redirect non-TOTP-enrolled users to an
enrollment page. Login and logout events are written to `audit_log`.
All 13 Category A floor tests pass.

## 3. Locked Decisions (do not re-litigate)

1. **User profiles live in `user_profiles`**, not
   `auth.users.user_metadata`. Supabase Auth owns identity; we own
   profile.
2. **Invitation flow uses `org_invitations`** with hashed tokens and
   7-day expiry. Email delivery out of scope — token returned in API
   response.
3. **Membership lifecycle via `status` column.** `removed` is
   soft-remove — row stays for audit trail.
4. **MFA enforcement**: enable TOTP in `supabase/config.toml`;
   middleware checks org `mfa_required` + user TOTP enrollment.
5. **Login/logout audit events** written with action keys
   `auth.login` and `auth.logout`.
6. **Org owner**: `is_org_owner boolean` on `memberships` with
   partial unique index. Cannot be suspended/removed/demoted.
7. **ActionName convention from 1.5A applies**: imperative-verb
   permission keys, past-tense audit action keys.

### 4.1 Migration 112 — `user_profiles`

```sql
-- =============================================================
-- 20240112000000_user_profiles.sql
-- Phase 1.5B: user profile table (identity owned by Supabase Auth,
-- profile owned by application).
-- =============================================================

BEGIN;

CREATE TABLE user_profiles (
  user_id               uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  first_name            text,
  last_name             text,
  display_name          text,
  avatar_storage_path   text,
  phone                 text,
  phone_country_code    text,
  preferred_locale      text,
  preferred_timezone    text,
  last_login_at         timestamptz,
  created_at            timestamptz NOT NULL DEFAULT now(),

  CONSTRAINT profile_phone_country_code_shape
    CHECK (phone_country_code IS NULL OR phone_country_code ~ '^\+[0-9]{1,3}$')
);

ALTER TABLE user_profiles ENABLE ROW LEVEL SECURITY;

-- Users see their own profile.
CREATE POLICY user_profiles_select_own ON user_profiles
  FOR SELECT USING (user_id = auth.uid());

-- Controllers see profiles of members in their orgs (for user list view).
CREATE POLICY user_profiles_select_org ON user_profiles
  FOR SELECT USING (
    EXISTS (
      SELECT 1 FROM memberships m
      WHERE m.user_id = user_profiles.user_id
        AND user_is_controller(m.org_id)
    )
  );

-- No INSERT/UPDATE/DELETE policies — profiles are created and
-- updated by getOrCreateProfile()/updateProfile() via adminClient.

COMMENT ON TABLE user_profiles IS
  'Application-owned user profile. Supabase Auth owns identity (email, password, MFA); we own display info, preferences, and login tracking. Auto-created on first login via getOrCreateProfile().';

COMMIT;
```

#### 4.1.1 Named CHECKs

- **`profile_phone_country_code_shape`** — same regex as
  `org_country_phone_code_shape` on `organizations`. Local
  consistency.

#### 4.1.2 `display_name` fallback

`display_name` is nullable in the schema. The **service layer**
implements the fallback chain: `display_name ?? (first_name + ' '
+ last_name).trim() ?? email`. The DB stores whatever the user set;
the service computes the display string on read. This keeps the DB
simple and the fallback logic in one place.

### 4.2 Migration 113 — `memberships` extension

```sql
-- =============================================================
-- 20240113000000_extend_memberships.sql
-- Phase 1.5B: membership lifecycle (status, owner, invited_via).
-- =============================================================
-- BLAST RADIUS: memberships is read by 8 files in src/ and tests/.
-- Every file that queries memberships without a status filter is a
-- potential security bug after this migration. The list:
--
-- 1. src/services/middleware/serviceContext.ts (buildServiceContext)
-- 2. src/services/auth/canUserPerformAction.ts
-- 3. src/services/auth/getMembership.ts
-- 4. src/services/org/membershipService.ts (listForUser)
-- 5. src/components/bridge/OrgSwitcher.tsx (browser client)
-- 6. supabase/migrations/20240101000000_initial_schema.sql
--    → user_has_org_access() and user_is_controller() SQL functions
--
-- This migration updates the two SQL helper functions. The four
-- TypeScript files must be updated in the service code step.
-- OrgSwitcher.tsx is a browser-client read that goes through RLS;
-- since user_has_org_access() now filters to active, the browser
-- query's visibility is automatically scoped to active memberships.
-- =============================================================

BEGIN;

CREATE TYPE membership_status AS ENUM (
  'active',
  'invited',
  'suspended',
  'removed'
);

ALTER TABLE memberships
  ADD COLUMN status           membership_status NOT NULL DEFAULT 'active',
  ADD COLUMN invited_via      uuid,
  ADD COLUMN is_org_owner     boolean NOT NULL DEFAULT false,
  ADD COLUMN suspended_at     timestamptz,
  ADD COLUMN suspended_by     uuid REFERENCES auth.users(id),
  ADD COLUMN removed_at       timestamptz,
  ADD COLUMN removed_by       uuid REFERENCES auth.users(id);

-- invited_via FK added after org_invitations table exists
-- (migration 114). For now it's a plain uuid column; migration 114
-- adds the FK constraint via ALTER TABLE.

-- Backfill: all existing rows are active.
-- (DEFAULT 'active' handles the backfill for the status column.)

-- Partial unique index: at most one owner per org.
CREATE UNIQUE INDEX idx_memberships_org_owner
  ON memberships (org_id)
  WHERE is_org_owner = true;

-- Backfill: the controller seed user for each org becomes the owner.
-- This UPDATE targets the dev-seed memberships. In production, the
-- first org creator is auto-set as owner by createOrgWithTemplate.
UPDATE memberships SET is_org_owner = true
WHERE role = 'controller'
  AND status = 'active'
  AND org_id IN (
    SELECT org_id FROM memberships
    WHERE role = 'controller' AND status = 'active'
    GROUP BY org_id
    HAVING COUNT(*) = 1
  );

-- Named CHECK: owner must be controller.
ALTER TABLE memberships
  ADD CONSTRAINT membership_owner_must_be_controller
    CHECK (NOT is_org_owner OR role = 'controller');

-- -----------------------------------------------------------------
-- UPDATE RLS HELPER FUNCTIONS to filter status = 'active'.
-- These are CREATE OR REPLACE — they overwrite the Phase 1.1
-- definitions from migration 001. The change is additive: one new
-- AND clause. Both functions keep their SECURITY DEFINER, STABLE,
-- SET search_path = '' posture.
-- -----------------------------------------------------------------

CREATE OR REPLACE FUNCTION public.user_has_org_access(target_org_id uuid)
RETURNS boolean
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = ''
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.memberships
    WHERE user_id = auth.uid()
      AND org_id = target_org_id
      AND status = 'active'
  );
$$;

CREATE OR REPLACE FUNCTION public.user_is_controller(target_org_id uuid)
RETURNS boolean
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = ''
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.memberships
    WHERE user_id = auth.uid()
      AND org_id = target_org_id
      AND role = 'controller'
      AND status = 'active'
  );
$$;

COMMIT;
```

#### 4.2.2 Named CHECKs

- **`membership_owner_must_be_controller`** —
  `NOT is_org_owner OR role = 'controller'`. An org owner must have
  `role = 'controller'`. This prevents the race where
  `changeUserRole` demotes an owner to `ap_specialist` without
  first transferring ownership. The CHECK fires regardless of the
  service-layer guard.

#### 4.2.3 `is_org_owner` protection rules (service-layer, not DB)

These are enforced by `membershipService` functions, not by DB
constraints:

- **Cannot remove the org owner.** `removeUser` rejects with
  `OWNER_CANNOT_BE_REMOVED` if target `is_org_owner = true`.
- **Cannot suspend the org owner.** `suspendUser` rejects with
  `OWNER_CANNOT_BE_SUSPENDED`.
- **Cannot change the org owner's role away from controller.**
  `changeUserRole` rejects with `OWNER_ROLE_CHANGE_DENIED` if
  target `is_org_owner = true` and `new_role != 'controller'`.
- **Ownership transfer** is not in 1.5B scope. If needed, a future
  `transferOwnership({ org_id, new_owner_user_id })` function
  atomically flips `is_org_owner` on both memberships in a
  transaction. Document this as OQ-01.

### 4.3 Migration 114 — `org_invitations`

```sql
-- =============================================================
-- 20240114000000_org_invitations.sql
-- Phase 1.5B: invitation-based org membership onboarding.
-- =============================================================

BEGIN;

CREATE TYPE invitation_status AS ENUM (
  'pending',
  'accepted',
  'expired',
  'revoked'
);

CREATE TABLE org_invitations (
  invitation_id        uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id               uuid NOT NULL REFERENCES organizations(org_id) ON DELETE CASCADE,
  invited_email        text NOT NULL,
  invited_by_user_id   uuid NOT NULL REFERENCES auth.users(id),
  role                 user_role NOT NULL,
  token_hash           text NOT NULL,
  expires_at           timestamptz NOT NULL DEFAULT (now() + interval '7 days'),
  accepted_at          timestamptz,
  accepted_by_user_id  uuid REFERENCES auth.users(id),
  status               invitation_status NOT NULL DEFAULT 'pending',
  created_at           timestamptz NOT NULL DEFAULT now(),

  -- Email is always stored lowercase.
  CONSTRAINT invitation_email_lowercase
    CHECK (invited_email = lower(invited_email)),
  -- Accepted invitations must carry the acceptor and timestamp.
  CONSTRAINT invitation_accepted_consistency
    CHECK (
      status != 'accepted'
      OR (accepted_at IS NOT NULL AND accepted_by_user_id IS NOT NULL)
    )
);

-- Only one pending invitation per email per org.
CREATE UNIQUE INDEX idx_invitation_pending_email
  ON org_invitations (org_id, invited_email)
  WHERE status = 'pending';

-- Supporting index for list-by-org queries.
CREATE INDEX idx_invitations_org_status
  ON org_invitations (org_id, status);

ALTER TABLE org_invitations ENABLE ROW LEVEL SECURITY;

-- Controllers see all invitations for their orgs.
CREATE POLICY org_invitations_select ON org_invitations
  FOR SELECT USING (user_is_controller(org_id));

-- No INSERT/UPDATE/DELETE policies — all mutations go through
-- invitationService via adminClient.

-- Now add the FK on memberships.invited_via → org_invitations.
ALTER TABLE memberships
  ADD CONSTRAINT memberships_invited_via_fkey
    FOREIGN KEY (invited_via) REFERENCES org_invitations(invitation_id);

COMMIT;
```

#### 4.3.1 Named CHECKs

- **`invitation_email_lowercase`** — `invited_email =
  lower(invited_email)`. Prevents mixed-case emails from creating
  duplicate-invite confusion.
- **`invitation_accepted_consistency`** — an accepted invitation
  must carry `accepted_at` and `accepted_by_user_id`. The service
  sets all three atomically.

#### 4.3.2 Invitation token flow

1. **`inviteUser`** generates a 32-byte random token
   (`crypto.randomBytes(32).toString('hex')`), then constructs
   the composite token: `{invitation_id}:{random_hex}`.
2. The **full composite token** is bcrypt-hashed
   (`await bcryptjs.hash(compositeToken, 10)`). Only the hash is
   stored in `org_invitations.token_hash`. The plaintext is never
   persisted.
3. The composite plaintext token is returned to the caller in the
   API response: `{ invitation_id, token }`. The caller shares
   the token with the invitee out-of-band (copy-paste, IM, email
   in Phase 2).
4. **`acceptInvitation`** receives `{ token }` in the request body.
   The service splits the token on the first `:` to extract the
   `invitation_id` prefix. It then does a **direct PK lookup**:
   `SELECT ... WHERE invitation_id = $1 AND status = 'pending'
   AND expires_at > now()`. If found, bcrypt-compare the full
   token against the stored `token_hash`. This is **O(1)** — one
   DB lookup + one bcrypt compare, not an O(n) scan.
5. If the bcrypt comparison fails or no pending invitation is
   found → `INVITATION_INVALID_OR_EXPIRED`.
6. **Rate limiting**: not rate-limited in 1.5B beyond Supabase
   Auth's default. If brute force becomes a concern, add a
   rate-limit in 1.5C or Phase 2.

### 5.1 `userProfileService`

```ts
// src/services/user/userProfileService.ts

export const userProfileService = {
  async getOrCreateProfile(
    input: { user_id: string; email: string },
    ctx: ServiceContext,
  ): Promise<UserProfile> { /* ... */ },

  async getProfile(
    input: { user_id: string },
    ctx: ServiceContext,
  ): Promise<UserProfile> { /* ... */ },

  async updateProfile(
    input: { user_id: string; patch: UpdateUserProfilePatch },
    ctx: ServiceContext,
  ): Promise<{ user_id: string; fields_changed: string[] }> { /* ... */ },
};
```

**`getOrCreateProfile`** is called from `buildServiceContext` (or
the sign-in page) on every authenticated request. It does an
UPSERT: if the `user_profiles` row exists, return it. If not,
insert a minimal row (`user_id`, `last_login_at = now()`) and
return it. This function is **not** audit-logged (it fires on
every request — logging it would flood the audit table).

**`updateProfile`** is audit-logged with `user.profile_updated`
and full `before_state`. The user can only update their own profile
(enforced at the route layer — the route reads `user_id` from
`ctx.caller.user_id`, not from the URL). Controllers cannot edit
other users' profiles.

**`getProfile`** is a read. Not `withInvariants`-wrapped per
OQ-07.

#### 5.1.1 `last_login_at` update mechanism

`getOrCreateProfile` updates `last_login_at = now()` on every
call. Since `buildServiceContext` calls `getOrCreateProfile` on
every request, this effectively becomes "last request" rather than
"last login." To make it truly "last login," limit the update to
the sign-in page's server action — call `getOrCreateProfile` from
the sign-in callback only, and call a lighter `getProfile` (no
upsert, no timestamp update) from `buildServiceContext`.

**Resolved (OQ-03, 2026-04-15):** sign-in only. `getOrCreateProfile`
is called from the sign-in callback; `buildServiceContext` calls
the lighter `getProfile` (no upsert, no timestamp update).

### 5.2 `invitationService`

```ts
// src/services/org/invitationService.ts

export const invitationService = {
  async inviteUser(
    input: { org_id: string; email: string; role: UserRole },
    ctx: ServiceContext,
  ): Promise<{ invitation_id: string; token: string }> { /* ... */ },

  async acceptInvitation(
    input: { token: string },
    ctx: ServiceContext,
  ): Promise<{ org_id: string; membership_id: string }> { /* ... */ },

  async revokeInvitation(
    input: { org_id: string; invitation_id: string },
    ctx: ServiceContext,
  ): Promise<{ invitation_id: string }> { /* ... */ },

  async resendInvitation(
    input: { org_id: string; invitation_id: string },
    ctx: ServiceContext,
  ): Promise<{ invitation_id: string; token: string }> { /* ... */ },

  async listPendingInvitations(
    input: { org_id: string },
    ctx: ServiceContext,
  ): Promise<{ invitations: OrgInvitation[] }> { /* ... */ },
};
```

**`inviteUser`** (controller-only via `withInvariants({ action:
'user.invite' })`):
1. Validate email is not already an active member of the org.
2. Generate 32-byte random token.
3. Bcrypt-hash the token (cost 10).
4. Insert `org_invitations` row with hash + `status = 'pending'`.
5. Create a `memberships` row with `status = 'invited'`,
   `invited_via = invitation_id`, `role = input.role`.
6. Audit-log `user.invited` with `entity_type = 'org_invitation'`.
7. Return `{ invitation_id, token }` (plaintext).

**Rejection branches:**
- `USER_ALREADY_MEMBER` — email already has an active membership.
- `INVITATION_ALREADY_PENDING` — email already has a pending
  invitation (partial unique index would catch this too, but
  typed error is cleaner).
- `INVITATION_WRITE_FAILED` — DB error.

**`acceptInvitation`**:
1. Split the provided token on first `:` to extract
   `invitation_id` and `full_token`.
2. Direct PK lookup: `SELECT ... WHERE invitation_id = $1
   AND status = 'pending' AND expires_at > now()`.
3. Bcrypt-compare `full_token` against `token_hash`.
4. If no row or bcrypt mismatch → `INVITATION_INVALID_OR_EXPIRED`.
5. Verify `invited_email = lower(ctx.caller.email)` — the
   invitation must be for this user's email.
6. In a single logical transaction:
   - Update `org_invitations` row: `status = 'accepted'`,
     `accepted_at = now()`, `accepted_by_user_id = caller`.
   - Update the `memberships` row (created in step 5 of
     `inviteUser`): `status = 'active'`.
6. Audit-log `user.invitation_accepted` with
   `entity_type = 'org_invitation'`.
7. Return `{ org_id, membership_id }`.

**`revokeInvitation`** (controller-only via `withInvariants({
action: 'user.invite' })`):
1. Update `status = 'revoked'` on the invitation.
2. Update `status = 'removed'` on the associated membership
   (the `invited` row created during `inviteUser`).
3. Audit-log `user.invitation_revoked`.

**`resendInvitation`** (controller-only):
1. Generate a new 32-byte token, hash it.
2. Update the invitation row: new `token_hash`, new
   `expires_at = now() + 7 days`.
3. Return `{ invitation_id, token }` (new plaintext).

**`listPendingInvitations`** is a read. Not wrapped.

**`changeUserRole`** (controller-only, `user.role.change`):

- Rejects `OWNER_CANNOT_BE_SUSPENDED` if target is org owner.

- Audit-log `user.suspended` with `before_state`.

- Sets `status = 'active'`, clears `suspended_at/by`.
- Audit-log `user.reactivated` with `before_state`.

**`removeUser`** (controller-only, `user.remove`):
- Rejects `OWNER_CANNOT_BE_REMOVED` if target is org owner.
- Soft-remove: `status = 'removed'`, `removed_at = now()`,
  `removed_by = ctx.caller.user_id`.
- Audit-log `user.removed` with `before_state`.

## 9. MFA Enforcement Middleware

### 9.1 Where in the middleware chain

Next.js `middleware.ts` (or a dedicated `src/middleware/mfaEnforcement.ts`
called from `middleware.ts`) runs before any route under
`/[locale]/[orgId]/...`. It does NOT run for:
- `/[locale]/sign-in` and `/[locale]/sign-out` (pre-auth)
- `/[locale]/mfa-enroll` (the enrollment target itself)
- `/api/auth/*` (auth endpoints must be accessible pre-MFA)
- `/api/industries` (no org scope)

### 9.2 What it checks

```
1. Extract orgId from the URL path.
2. If no orgId → skip (non-org-scoped route).
3. Load org from DB → check organizations.mfa_required.
4. If mfa_required = false → pass through.
5. If mfa_required = true:
   a. Read the user's session AMR (Authentication Methods Reference)
      from Supabase Auth: supabase.auth.mfa.getAuthenticatorAssuranceLevel().
   b. If currentLevel >= 'aal2' → pass through (TOTP verified).
   c. If currentLevel < 'aal2' → redirect to /[locale]/mfa-enroll.
```

### 9.3 The enrollment page

`src/app/[locale]/mfa-enroll/page.tsx` — client component that:
1. Calls `supabase.auth.mfa.enroll({ factorType: 'totp' })` to get
   a QR code URI.
2. Displays the QR code for the user to scan.
3. Accepts a TOTP code, calls `supabase.auth.mfa.challengeAndVerify`.
4. On success, redirects back to the intended org page.

Phase 1.5B ships the enrollment page as a functional stub — QR code
display + verification. No polish, no recovery codes UI (Phase 2).

### 9.4 Test approach

The MFA middleware is tested by:
- Integration test CA-09: Mock the AMR response, verify redirect
  to `/mfa-enroll` when org requires MFA and user lacks TOTP.
- The middleware itself is a thin function — most logic is in
  Supabase Auth's MFA API.

## 10. Forward-Compatibility Notes

### OQ-10 — Accountant-firm-as-org

The `org_invitations` table is forward-compatible with the
accountant-firm pattern. A firm user accepting an invitation to a
client org uses the same flow. When firm-as-org lands (Phase 2+),
the additions would be:
- `organizations.org_type` enum (`client`, `firm`)
- `memberships.access_expires_at timestamptz` for time-bounded
  access
- Entity-attached comments table

None of these require changes to 1.5B tables. The invitation flow,
membership lifecycle, and user profiles all work as-is for the
firm pattern.

### OQ-11 — Billing contact vs primary contact

With `user_profiles` and `is_org_owner`, the org owner's profile
serves as the primary contact. The billing contact can be modeled
as:
- `organizations.billing_contact_user_id uuid REFERENCES
  auth.users(id)` — a nullable FK added in Phase 2 alongside
  actual billing integration.
- Or a `memberships` tag (`is_billing_contact boolean`) — same
  pattern as `is_org_owner`.

The 1.5B schema doesn't need changes to support either path.

## 14. Open Questions (consolidated)

### OQ-01 — Ownership transfer — RESOLVED (2026-04-15)

**Description.** No `transferOwnership` function in 1.5B. If the
org owner leaves, there's no mechanism to transfer.

**Resolution (2026-04-15).** Deferred to 1.5C or Phase 2. The
owner protection rules prevent accidental removal. An explicit
transfer function is a low-frequency operation.

**Original rationale.** Defer to 1.5C or Phase 2.

### OQ-02 — Invitation acceptance: token-embedded invitation_id — RESOLVED (2026-04-15)

**Description.** Should `acceptInvitation` find the invitation by
email-based scan or direct PK lookup via a token-embedded id?

**Resolution (2026-04-15).** Token format is
`{invitation_id}:{random_bytes}`. Store bcrypt of the full token
string. On accept: split token on first `:`, use the
`invitation_id` prefix for a PK lookup, then bcrypt-compare the
full token against the stored hash. This is O(1) vs O(n) email
scan. The §4.3.2 token flow is updated accordingly.

### OQ-03 — `last_login_at` update frequency — RESOLVED (2026-04-15)

**Description.** Should `last_login_at` update on every request
(noisy writes) or only on sign-in (accurate semantics)?

**Resolution (2026-04-15).** Sign-in only. Call
`getOrCreateProfile` from the sign-in callback;
`buildServiceContext` uses the lighter `getProfile` (no upsert,
no timestamp update).

### OQ-04 — `audit_log.org_id` nullability for auth events — RESOLVED (2026-04-15)

**Description.** Login/logout events are not org-scoped, but
`audit_log.org_id` is currently NOT NULL. Need to either make it
nullable or pick a workaround.

**Resolution (2026-04-15).** Make `org_id` nullable via ALTER in
migration 113. Login/logout audit rows carry `org_id = NULL`. This
is the cleanest option — a pseudo-org-id would pollute queries,
and per-org rows for login would be misleading.

### OQ-05 — bcrypt dependency choice — RESOLVED (2026-04-15)

**Description.** `bcrypt` (native C binding, faster) vs `bcryptjs`
(pure JS, no native build step).

**Resolution (2026-04-15).** Use `bcryptjs` — no native build
step, simpler CI. Invitation acceptance is a low-frequency
operation (cost 10 hash takes ~100ms in pure JS, acceptable).

### OQ-06 — Expired invitation cleanup — RESOLVED (2026-04-15)

**Description.** Pending invitations past `expires_at` should
eventually be marked `status = 'expired'`.

**Resolution (2026-04-15).** Lazy expiration — filter
`expires_at > now()` at query time in both `acceptInvitation` and
`listPendingInvitations`. No cron needed in 1.5B. Phase 2 adds
cleanup if data hygiene becomes a concern.
