*This brief is the execution spec for Phase 1.5C of The Bridge.
Phase 1.5A (org profile) and 1.5B (users/invites/MFA) are complete.
Phase 1.5C replaces the Phase 1.1 role-based TypeScript permission
map with a table-driven permission catalog. This is the third and
final sub-phase of the 1.5 split.*

## 1. Goal

**Phase 1.5C replaces the static `ROLE_PERMISSIONS` TypeScript map
with a table-driven permission catalog (`roles`, `permissions`,
`role_permissions`) so Phase 1.2 agent integration and Phase 2
org-custom-roles have a queryable permission surface to build
against.**

What "done" means in one paragraph: three new tables exist and are
seeded with the 16 current permission keys mapped to three system
roles (matching the existing `ROLE_PERMISSIONS` map exactly).
`canUserPerformAction` is rewritten from a TypeScript map lookup
to a SQL query against `role_permissions`. A new SQL helper
`user_has_permission(org_id, permission_key)` is available for
future RLS policies. `memberships.role_id` is populated via
two-step backfill from the legacy `role` enum. The `ActionName`
TypeScript union is hand-maintained with a parity test asserting
set-equality against the `permissions` table. Every existing
`withInvariants` call site works unchanged. All 12 Category A
floor tests pass. Zero regressions on the existing 136 tests.

### 4.1 Migration 116 — Permission catalog

```sql
-- =============================================================
-- 20240116000000_permission_catalog.sql
-- Phase 1.5C: table-driven permission catalog
-- =============================================================
-- Replaces the ROLE_PERMISSIONS TypeScript map in
-- canUserPerformAction.ts. After this migration:
-- - roles table: 3 system rows (executive, controller, ap_specialist)
-- - permissions table: 16 rows (one per ActionName value)
-- - role_permissions: join table mapping roles → permissions
-- - user_has_permission(): new SQL helper for future RLS policies
--
-- The TypeScript ActionName union is KEPT as a hand-maintained
-- catalog. A parity test asserts set-equality with the permissions
-- table. See docs/09_briefs/phase-1.5/1.5C-brief.md §7.
-- =============================================================

BEGIN;

-- -----------------------------------------------------------------
-- ROLES
-- -----------------------------------------------------------------

CREATE TABLE roles (
  role_id       uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  role_key      text UNIQUE NOT NULL,
  display_name  text NOT NULL,
  description   text,
  is_system     boolean NOT NULL DEFAULT true,
  org_id        uuid REFERENCES organizations(org_id),
  sort_order    integer NOT NULL DEFAULT 0,
  created_at    timestamptz NOT NULL DEFAULT now(),

  CONSTRAINT role_system_org_consistency
    CHECK (
      (is_system = true AND org_id IS NULL) OR
      (is_system = false AND org_id IS NOT NULL)
    )
);

CREATE INDEX idx_roles_org ON roles (org_id) WHERE org_id IS NOT NULL;

ALTER TABLE roles ENABLE ROW LEVEL SECURITY;

CREATE POLICY roles_select ON roles
  FOR SELECT TO authenticated USING (
    is_system = true OR user_has_org_access(org_id)
  );

-- -----------------------------------------------------------------
-- PERMISSIONS
-- -----------------------------------------------------------------

CREATE TABLE permissions (
  permission_key  text PRIMARY KEY,
  display_name    text NOT NULL,
  description     text,
  category        text NOT NULL,
  sort_order      integer NOT NULL DEFAULT 0,
  created_at      timestamptz NOT NULL DEFAULT now()
);

ALTER TABLE permissions ENABLE ROW LEVEL SECURITY;

CREATE POLICY permissions_select ON permissions
  FOR SELECT TO authenticated USING (true);

-- -----------------------------------------------------------------
-- ROLE_PERMISSIONS (join)
-- -----------------------------------------------------------------

CREATE TABLE role_permissions (
  role_id         uuid NOT NULL REFERENCES roles(role_id) ON DELETE CASCADE,
  permission_key  text NOT NULL REFERENCES permissions(permission_key) ON DELETE CASCADE,
  granted_at      timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (role_id, permission_key)
);

ALTER TABLE role_permissions ENABLE ROW LEVEL SECURITY;

CREATE POLICY role_permissions_select ON role_permissions
  FOR SELECT TO authenticated USING (true);

  ('org.create',             'Create organizations',      'Organization',  50),
  ('audit_log.read',         'View audit log',            'Reports',       60),
  ('ai_actions.read',        'View AI actions',           'Agent',         70),
  ('org.profile.update',     'Update org profile',        'Organization',  80),
  ('org.address.create',     'Add org addresses',         'Organization',  90),
  ('org.address.update',     'Edit org addresses',        'Organization', 100),

  ('org.address.set_primary','Set primary address',       'Organization', 120),

  ('user.suspend',           'Suspend/reactivate users',  'Users',        150),
  ('user.remove',            'Remove users',              'Users',        160);

-- -----------------------------------------------------------------
-- SEED: role_permissions (matches ROLE_PERMISSIONS TypeScript map)
--
-- Controller: all 16
-- AP Specialist: journal_entry.post, chart_of_accounts.read, ai_actions.read
-- Executive: chart_of_accounts.read, audit_log.read, ai_actions.read

-- Controller — all permissions

FROM roles r, permissions p

  AND p.permission_key IN ('journal_entry.post', 'chart_of_accounts.read', 'ai_actions.read');

-- Executive — 3 permissions
INSERT INTO role_permissions (role_id, permission_key)
SELECT r.role_id, p.permission_key
FROM roles r, permissions p
WHERE r.role_key = 'executive'
  AND p.permission_key IN ('chart_of_accounts.read', 'audit_log.read', 'ai_actions.read');

-- -----------------------------------------------------------------
-- SQL HELPER: user_has_permission()
-- **Installed in migration 117, not 116** — the function body
-- references memberships.role_id which doesn't exist until 117.
-- Placed here in §4.1 for catalog completeness; see §4.2 for
-- the actual CREATE FUNCTION placement.
-- Same posture as user_has_org_access and user_is_controller.
-- Available for new RLS policies; existing policies are NOT
-- rewritten in 1.5C.
-- -----------------------------------------------------------------

CREATE OR REPLACE FUNCTION public.user_has_permission(
  target_org_id uuid,
  target_permission_key text
)
RETURNS boolean
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = ''
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.memberships m
    JOIN public.role_permissions rp ON rp.role_id = m.role_id
    WHERE m.user_id = auth.uid()
      AND m.org_id = target_org_id
      AND m.status = 'active'
      AND rp.permission_key = target_permission_key
  );
$$;

REVOKE ALL ON FUNCTION public.user_has_permission(uuid, text) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.user_has_permission(uuid, text) TO authenticated;

COMMIT;
```

#### 4.1.1 Named CHECKs

- **`role_system_org_consistency`** — `(is_system = true AND org_id
  IS NULL) OR (is_system = false AND org_id IS NOT NULL)`. System
  roles are global (no org scope); future custom roles are always
  org-scoped.

#### 4.1.2 Why CROSS JOIN for controller seed

The controller gets every permission. Using `CROSS JOIN` means new
permissions added in a future migration automatically need an
`INSERT INTO role_permissions` for the controller role — the CROSS
JOIN only fires once at seed time, not dynamically.

### 4.2 Migration 117 — `memberships.role_id`

```sql
-- =============================================================
-- 20240117000000_memberships_role_id.sql
-- Phase 1.5C: two-step role_id backfill on memberships
-- =============================================================
-- Same discipline as 1.5A industry_id: add nullable → backfill
-- from legacy enum via bridge → SET NOT NULL. Legacy role enum
-- column stays until follow-up migration.
--
-- BLAST RADIUS: every file that inserts into memberships needs
-- to write both role AND role_id during the cutover window.
--
--   src/services/org/orgService.ts (createOrgWithTemplate)
--   src/services/org/invitationService.ts (acceptInvitation)
--   src/services/org/membershipService.ts (changeUserRole)
--   src/db/seed/dev.sql
--   tests/integration/ownerPartialUnique.test.ts
--
-- The grep command:
--   grep -rn "from('memberships').*insert\|\.insert.*role" src/ tests/
-- =============================================================

BEGIN;

ALTER TABLE memberships ADD COLUMN role_id uuid REFERENCES roles(role_id);

-- Backfill via the role_key bridge.
UPDATE memberships m
SET role_id = r.role_id
FROM roles r
WHERE r.role_key = m.role::text
  AND r.is_system = true
  AND m.role_id IS NULL;

ALTER TABLE memberships ALTER COLUMN role_id SET NOT NULL;

CREATE INDEX idx_memberships_role_id ON memberships (role_id);

-- The existing membership_owner_must_be_controller CHECK uses the
-- legacy role column: NOT is_org_owner OR role = 'controller'.
-- During the cutover window both columns are populated, so this
-- CHECK continues to fire correctly. Adding a second CHECK on
-- role_id would be redundant — the backfill guarantees role_id
-- matches role. When the legacy column is dropped in the follow-up
-- migration, the CHECK is replaced with:
--   NOT is_org_owner OR role_id = (SELECT role_id FROM roles WHERE role_key = 'controller')
-- For now, leave the existing CHECK as-is.

COMMIT;
```

#### 4.2.1 Blast-radius file list (cutover writes)

Every file that inserts or updates `memberships.role` must also
write `role_id` during the cutover window. Resolved via a
`roles.role_key` lookup subquery or a cached `role_id`:

| File | What changes |
|---|---|
| `src/services/org/orgService.ts` | `createOrgWithTemplate`: add `role_id` to membership insert (lookup by `role_key = 'controller'`) |
| `src/services/org/invitationService.ts` | `acceptInvitation`: add `role_id` to membership insert (lookup by invitation's `role`) |
| `src/services/org/membershipService.ts` | `changeUserRole`: update both `role` and `role_id` on role change |

#### 4.2.2 `membership_owner_must_be_controller` CHECK

**Decision:** keep the existing CHECK on the legacy `role` column.
It continues to fire correctly because both columns stay populated
during the cutover. Adding a second CHECK on `role_id` is
redundant since the backfill guarantees parity. When the legacy
column is dropped, replace the CHECK with a `role_id`-based
version. Not an open question — documented here for the follow-up
migration author.

## 5. `canUserPerformAction` Rewrite

### 5.1 Signature (unchanged)

```ts
export async function canUserPerformAction(
  ctx: ServiceContext,
  action: ActionName,
  orgId: string,
): Promise<AuthorizationResult>
```

### 5.2 Implementation (SQL query replaces TypeScript map)

```ts
const db = adminClient();

const { data: membership, error: memErr } = await db
  .from('memberships')
  .select('role_id')
  .eq('user_id', ctx.caller.user_id)
  .eq('org_id', orgId)
  .eq('status', 'active')
  .maybeSingle();

if (memErr) {
  return { permitted: false, reason: `Membership lookup failed: ${memErr.message}` };
}
if (!membership) {
  return { permitted: false, reason: `No membership for user in org_id=${orgId}` };
}

const { data: perm, error: permErr } = await db
  .from('role_permissions')
  .select('permission_key')
  .eq('role_id', membership.role_id)
  .eq('permission_key', action)
  .maybeSingle();

if (permErr) {
  return { permitted: false, reason: `Permission lookup failed: ${permErr.message}` };
}

if (!perm) {
  return { permitted: false, reason: `Role does not have permission '${action}'` };
}

return { permitted: true, reason: 'ok' };
```

### 5.3 Why two queries, not PostgREST embed

PostgREST's embed syntax (`memberships!inner(...)`) requires a
direct FK between `memberships` and `role_permissions`. The FK
path is `memberships.role_id → roles.role_id` and
`role_permissions.role_id → roles.role_id` — but PostgREST cannot
join `memberships` to `role_permissions` through `roles` without
an explicit view or RPC. The two-query approach is:
- Same number of DB round-trips as the current implementation
  (which does one membership query + one TypeScript map lookup,
  now replaced by one membership query + one `role_permissions`
  query).
- Avoids adding an RPC function just for the service layer (the
  `user_has_permission` SQL helper exists for RLS, not for
  TypeScript).
- Keeps the error reporting granular (separate "no membership" vs
  "no permission" messages).

### 5.4 What is removed

The `ROLE_PERMISSIONS` constant (the TypeScript `Record<UserRole,
ReadonlySet<ActionName>>`) is deleted entirely. The `UserRole`
type export from `canUserPerformAction.ts` is kept (still used by
`getMembership.ts` and `OrgSwitcher.tsx` for display purposes
during the cutover window). Phase 2 deprecates the type in favor
of `roles.role_key`.

## 7. TypeScript ↔ DB Parity Test

```ts
// tests/integration/permissionParity.test.ts

it('ActionName union matches permissions table', async () => {
  const db = adminClient();
  const { data: dbPerms } = await db
    .from('permissions')
    .select('permission_key');
  const dbKeys = new Set(
    (dbPerms ?? []).map((p: { permission_key: string }) => p.permission_key),
  );

  // Import the ActionName values from the source of truth.
  // This list is hand-maintained; adding a new ActionName requires
  // adding a corresponding permissions row in a migration.
  const tsKeys: Set<string> = new Set([
    'journal_entry.post', 'chart_of_accounts.read',
    'chart_of_accounts.write', 'period.lock', 'org.create',
    'audit_log.read', 'ai_actions.read', 'org.profile.update',
    'org.address.create', 'org.address.update', 'org.address.delete',
    'org.address.set_primary', 'user.invite', 'user.role.change',
    'user.suspend', 'user.remove',
  ]);

  // Symmetric difference must be empty.
  const inDbNotTs = [...dbKeys].filter((k) => !tsKeys.has(k));
  const inTsNotDb = [...tsKeys].filter((k) => !dbKeys.has(k));

  expect(inDbNotTs).toEqual([]);
  expect(inTsNotDb).toEqual([]);
});
```

This test catches drift in both directions: a new `ActionName`
added without a `permissions` row, or a row seeded without a
matching TypeScript union member.

## 9. Forward Compatibility (Phase 2)

### 9.1 Org-custom roles

The `is_system = false` path on `roles` supports Phase 2 custom
roles. A Phase 2 migration:
1. Adds a UI for creating org-scoped roles.
2. Adds a service `roleService.createCustomRole(org_id, ...)`.
3. The `role_system_org_consistency` CHECK ensures custom roles
   always carry an `org_id`.
4. The `roles_select` RLS policy already shows org-scoped roles
   only to members of that org.

### 9.2 Module feature gating

Permission categories (`'Accounting'`, `'Organization'`, etc.) are
forward-compat with module toggles. A Phase 2 `org_modules` table
could gate which categories are visible to an org. 1.5C does not
build this — the categories exist for UI grouping only.

### 9.3 Existing RLS policy migration

The `user_has_permission()` helper is available. Phase 2 can
replace `user_is_controller(org_id)` calls in policies like
`fiscal_periods_insert` with
`user_has_permission(org_id, 'period.lock')` — more precise,
permission-based. 1.5C does not touch existing policies.

## 10. API Changes

No new API routes in 1.5C. The existing routes continue to call

Two optional read-only endpoints for the future UI phase:

- `GET /api/roles/[roleId]/permissions` — list permissions for a
  role

These are **deferred to Phase 1.5D or Phase 2** when the role

## 11. Open Questions

### OQ-01 — `memberships.role` column drop timing

**Description.** When does the follow-up migration that drops the
legacy `role` enum column land?

**Resolution.** Same criteria as `organizations.industry` from
1.5A: (a) every service function uses `role_id`, (b) `types.ts`
regenerated, (c) one full deployment cycle. The `user_role` enum
type itself stays until `org_invitations.role` is also migrated
(Phase 1.6 or Phase 2).

### OQ-02 — `org_invitations.role` migration

**Description.** `org_invitations.role` uses the `user_role` enum
directly. Migrating it to `role_id uuid REFERENCES roles` is a
separate column migration. Not in 1.5C scope — the column works
as-is because the three enum values match the three
`roles.role_key` values.

**Deferral target.** Phase 1.6 or Phase 2 alongside the
`memberships.role` column drop.
